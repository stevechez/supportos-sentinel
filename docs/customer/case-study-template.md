# Case Study Template (Phase 20G)

Fill in from real evidence only -- every section should trace back to a
specific entry in `docs/customer/value-evidence-log.md` or a specific
interview response in `docs/customer/interview-template.md`. If a section
can't be filled in with something real yet, leave it blank rather than
writing something plausible-sounding. A case study with one honest, small
result is more useful (and more credible) than one with an invented number.

## Problem

possible, pulled directly from `docs/customer/value-evidence-log.md`'s

"Biggest recurring issues" sections. Not a generic industry problem

## Discovery

What Sentinel actually found for them -- name the specific finding (e.g.
"a recurring password-reset issue affecting 12 conversations over one
week"), not a generic capability description ("Sentinel detects patterns").
Pull from their actual first-insight data or a later finding, whichever was
the one that mattered to them (per the interview's "Most valuable insight"
section).

## Improvement

What changed as a direct result -- a specific action (a documentation edit,
a macro change, a process tweak), sourced from
`docs/customer/value-evidence-log.md`'s "What actually changed" field.
Not "they started using Sentinel more" -- an actual change to how they
operate.

## Outcome

Why it mattered to them, specifically -- their own framing of the value
(time saved, escalations reduced, faster response, whatever they actually
said), not a generalized ROI claim. If they gave a number, use their
number and attribute it to them explicitly ("the customer estimated...").
Never calculate or infer a number they didn't state themselves.
